# Mnemo UI and UX Audit

Date: 2026-07-10

Status: Gate A evidence review. No production UI implementation is included in this phase.

## Executive summary

Mnemo already has the right product foundation: a native two-tab shell, semantic SwiftUI typography, opaque content surfaces, explicit local-first language, safe archive/delete flows, and source cards that open the canonical saved memory. The redesign should refine this foundation rather than replace it.

The highest-impact issues are structural rather than cosmetic:

1. Capture is presented through three competing interaction models.
2. Onboarding and capture layouts can clip at large Dynamic Type; onboarding text is visibly truncated even at the default size.
3. Liquid Glass usage is screen-local, layered incorrectly, and has no Reduce Transparency or Increased Contrast policy.
4. Source access stops being actionable after three citations.
5. Browse filters combine unrelated dimensions into one exclusive chip rail.
6. Memory Detail gives internal processing metadata more visual weight than the original capture.
7. Settings and several content screens overuse separate rounded cards, flattening hierarchy.
8. The production AppIcon and AccentColor asset entries contain no artwork or color values.
9. Important state changes are not announced to VoiceOver.
10. The current launch splash is tied to initialization duration and its intended reveal animation is dead code.

The recommended direction is conservative at the navigation level and substantive at the component level: retain native Recall/Chat and Memories/Browse tabs, consolidate capture into one contextual entry per screen, rebuild shared surface and accessibility policies first, then redesign Chat and source evidence as the flagship experience.

## Method

The audit included:

- all app-owned SwiftUI `View` declarations under `MnemoApp/Mnemo` and `MnemoUI/Sources`
- app shell, navigation, sheets, toolbar composition, and accessibility identifiers
- uses of design tokens, literal frames, shadows, materials, glass, animations, line limits, and custom controls
- launch configuration, app icon assets, accent color assets, and the custom Mnemo mark
- simulator build and direct screenshot inspection in light and dark appearance
- runtime accessibility-hierarchy inspection with XcodeBuildMCP
- prior Mnemo design and validation documents for unresolved findings
- current official Apple HIG, SwiftUI, accessibility, Liquid Glass, launch, and app-icon guidance

Dependency source, generated files, package examples, and non-UI business logic were excluded except where needed to establish UI state and behavior boundaries.

## Current strengths

- Native `TabView`, `NavigationStack`, `List`, `searchable`, sheets, menus, and confirmation dialogs are already the default.
- Chat is correctly the primary product surface and distinguishes saving a memory from asking about one.
- Local/private positioning is direct and avoids cloud-AI claims.
- Source cards open the canonical SwiftData record and preserve archive/delete safety.
- Most text uses semantic system styles and therefore inherits Dynamic Type and Bold Text behavior.
- Most icon-only navigation controls have explicit labels and 44-point frames.
- Reduce Motion is considered in many custom transitions.
- Memory content is generally placed on opaque surfaces rather than widespread glass.
- Browse uses a single-column list, native search, and safe swipe actions.
- App Lock wording is calm and uses the available biometric method.
- AI Lab and smoke-test controls do not leak into Release.
- The existing code-native mnemonic-thread mark is distinctive, monochrome-capable, and aligned with memory/continuity.

## Cross-app findings

### 1. Capture has no single interaction grammar

Capture appears as:

- a four-option action area on the empty Chat landing
- a plus menu plus a separate microphone button in the active Chat composer
- an expandable floating action button on Browse
- a second large `Write memory` button in empty Browse

The duplicated Browse actions are visible in the baseline screenshots. `NavigationCoordinator.Tab.capture` also exists but is unused. This creates unnecessary learning cost and makes capture controls compete with recall content.

Recommendation: preserve the native two-tab shell and use one contextual capture entry. On the Recall landing, show Write as primary and voice/camera/photo as secondary. In an active conversation, use one add menu in the composer. In Memories, use a native Add toolbar/menu control and remove the custom floating button.

### 2. Content hierarchy is flattened by repeated cards

Chat messages, recovery states, Browse rows, Memory Detail sections, onboarding information, Settings rows, and capture confirmations frequently rebuild rounded opaque cards with borders or shadows. When every section is elevated, primary memory content and secondary metadata look equally important.

Recommendation: reserve cards for repeated records and genuinely grouped controls. Use unframed page sections, native list grouping, dividers, and disclosure for metadata. Answers, original captures, and source evidence should receive stable surfaces; page sections should not each become floating cards.

### 3. Shared UI behavior is duplicated

Primary/secondary buttons, cards, capture layouts, status capsules, and empty states are repeatedly reconstructed. `DS.ComponentTokens.Card` exists but does not prevent screen-local implementations. Several surface tokens share identical values, and legacy names such as `brandSage` now alias indigo.

Recommendation: establish semantic surfaces, adaptive borders, shared button styles, capture scaffolding, source treatment, metadata layout, and accessibility-aware material policy in MnemoUI before screen migration.

### 4. No preview matrix exists

No `#Preview` declarations were found. Empty, long-content, error, dark, accessibility Dynamic Type, Increased Contrast, and Reduce Transparency states cannot be inspected independently.

Recommendation: introduce focused component/screen previews during implementation, backed by UI-only sample models that do not change production persistence or AI logic.

## Screen findings

### Chat

Strengths:

- capture and recall are explained separately
- multiline input is supported
- generation copy says `Looking through saved memories`, which describes observable work
- source cards open the canonical Memory Detail flow
- missing-source presentation exists

Issues:

- assistant text, source cards, timestamps, and shadows create excessive visual containers
- all message arrivals force an animated scroll to the bottom, even if the user is reading earlier content
- the toolbar action labeled `Home` actually clears the ephemeral conversation; `New conversation` is clearer
- Settings is available only from Chat
- the privacy badge applies interactive glass even though it is not interactive and forces white text
- the composer uses regular material with no Reduce Transparency fallback
- the active composer presents plus, microphone, text, and send as competing actions
- the send symbol transition is not gated by Reduce Motion
- timestamps use low-contrast tertiary text and add noise to every message

### Source cards

Strengths:

- evidence is visually separated from the assistant answer
- each rendered citation has a source label, summary, and accessible tap action
- the source opens the real saved memory

Issues:

- only the first three citations are actionable
- additional citations become noninteractive `more sources not shown` text
- if citation metadata is empty but cited IDs exist, the fallback is untappable
- color and a narrow capsule do too much of the visual differentiation
- source summaries are capped at four lines without an alternate compact expansion

Recommendation: keep all sources reachable, show a compact source summary connected to the answer, include a source icon/label as well as color, and preserve the existing Memory Detail destination.

### Browse

Strengths:

- single-column layout is appropriate for memory text
- native search and swipe actions are used
- type is communicated with an icon as well as color

Issues:

- eight chips mix type (`Preferences`, `Lists`, `Credentials`), subject (`People`), and lifecycle (`Active`, `Dormant`, `Review`) in one exclusive taxonomy
- the horizontal rail is hard to scan and extends offscreen
- empty Browse duplicates `Write memory` and the floating Add button
- 96-point minimum rows reduce density for large collections
- primary text is capped at four lines and metadata stays in a rigid row
- selected filters are differentiated mainly by color
- up to 13 list rows animate independently on appearance
- fixed FAB padding is not composed with safe-area/tab-bar layout

Recommendation: keep a list, retain native search, separate type and lifecycle filtering, use a menu or small number of scopes, and remove row-by-row entrance choreography.

### Memory Detail

Strengths:

- saved summary is presented first
- archive and permanent delete remain separate and confirmed
- long-form information is scrollable

Issues:

- processing tier, recall priority, and review status precede the original capture
- original source text is rendered as tertiary footnote content
- repeated cards and shadows give every section equal visual weight
- metadata rows assume a horizontal label/spacer/value layout

Recommendation: order content as summary, original capture, source/date, type/status, then disclosed provenance/review details. Render original capture as body text on a stable surface and allow metadata to stack at accessibility sizes.

### Text capture

Strengths:

- writing is the visual focus
- review-before-save behavior is clear
- multiline entry is supported

Issues:

- root and confirmation layouts do not scroll
- keyboard plus large text can displace the primary action
- save/review state changes are not announced
- button construction is duplicated

### Voice capture

Strengths:

- recording, transcript confirmation, and permission-denied states exist
- manual transcript fallback is available
- waveform motion is partially gated for Reduce Motion

Issues:

- no elapsed recording time is shown
- the red pulse and strong feedback read as alarm-like for a private memory tool
- root and confirmation states do not scroll
- several text-only actions do not guarantee a 44-point target
- recording/transcription transitions are not announced
- technical speech/simulator errors can leak into user-visible copy

### Image capture

Strengths:

- native camera and photo library paths are used
- the selected route is understandable

Issues:

- the camera path opens immediately rather than first establishing a consistent confirmation structure
- there is no explicit retake/choose-another action after OCR
- the clarifying note is single-line
- the fixed image preview plus keyboard can displace controls
- captured images lack an accessibility description
- OCR wording can imply more certainty than is appropriate

### Onboarding

Strengths:

- language is privacy-led and avoids exaggerated AI claims
- the core capture/ask/source story is present

Issues:

- four forward-only pages repeat the no-account message
- there is no Back action
- progress begins at zero while page one is visible
- the layout does not scroll
- feature descriptions have `lineLimit(2)` and are visibly truncated at default Dynamic Type in the baseline screenshot
- the progress bar has no accessibility label or value
- nested staggered animations add complexity without improving comprehension
- stale enum names no longer match their content

Recommendation: use three stages: Remember privately, Ask naturally, Verify the source. Add Back after the first stage, make the content scroll-adaptive, and request permissions only in context.

### App Lock and privacy shield

Strengths:

- the presentation is calm
- native Face ID/Touch ID/passcode language is used
- retry behavior exists
- the app-switcher privacy shield is conceptually appropriate

Issues:

- the explanatory card and separate primary button split the hierarchy
- lock/error transitions are not announced or focused for VoiceOver
- the visual card is larger and more theatrical than necessary

### Settings

Strengths:

- implementation uses a native `List`
- destructive actions and confirmations exist
- AI Lab is correctly DEBUG-only

Issues:

- Device, Privacy, and Security repeat local/no-account messaging
- three unavailable Mnemo Sense rows are visible in normal builds
- list rows are restyled as large separate cards, producing administrative density
- fixed status capsules can crowd large text
- `BackupRestoreView` creates another `NavigationStack` inside Settings navigation
- Settings is only reachable from Chat

Recommendation: consolidate into Privacy and Security, Memory and Backup, Data Management, and DEBUG diagnostics. Remove or defer inactive Sense rows. Preserve native list hierarchy and use disclosure instead of decorative cards.

### Launch experience

- The system launch screen is generated and static, with no custom launch artwork.
- `SplashView` blocks until initialization completes, including DEBUG backfill work.
- Its duration is workload-dependent rather than a controlled brand reveal.
- `appeared` starts as `true`, so the intended on-appear reveal does not run.

Recommendation: keep the system launch screen minimal and visually continuous with the first real frame. Default to no blocking logo reveal. Any post-launch transition must overlay ready content, never delay interaction, and remain a separate human review gate.

### DEBUG surfaces

- AI Lab and smoke-test interfaces are correctly conditional.
- No DEBUG diagnostics were found in Release UI paths.
- Technical language inside AI Lab is appropriate for its audience.

## Accessibility findings

### Dynamic Type and Bold Text

Highest-risk layouts:

- non-scrolling onboarding, text capture, voice capture, and image confirmation
- two-line onboarding descriptions
- fixed-height image buttons
- horizontal Memory Detail metadata
- fixed status capsules in Settings
- four capture controls in one recovery row
- rigid Browse source/date metadata rows

The default-size onboarding screenshot already shows truncated feature descriptions, so this is a current defect rather than a theoretical accessibility risk.

### VoiceOver

Key controls and rendered source cards have useful labels. Missing behaviors include announcements or live-region equivalents for:

- generation start/completion
- memory save success/error
- recording and transcription state
- archive/delete completion
- filter-result changes
- App Lock error/unlock state

`MemorySavedOverlay` disappears after about 1.08 seconds and does not explicitly move focus or announce completion.

### Contrast and color independence

The current color system adapts only to light/dark appearance. It does not read `colorSchemeContrast` or `accessibilityDifferentiateWithoutColor`.

Approximate contrast from coded sRGB values:

- light tertiary text on grouped canvas: about 3.2-3.4:1
- dark tertiary text on elevated surface: about 3.15:1
- dark tertiary text on grouped canvas: about 4.22:1
- white on the dark-mode accent: about 4.47:1

Tertiary text is used for dates, timestamps, and original-memory supporting text. It must either be noncritical only or gain stronger adaptive variants. Selected filters need a checkmark, label, or shape change in addition to color.

### Reduce Transparency and Increased Contrast

No app code reads:

- `accessibilityReduceTransparency`
- `colorSchemeContrast`
- `accessibilityDifferentiateWithoutColor`

The Chat composer always uses `.regularMaterial`, and the iOS 26 privacy badge/FAB always apply custom glass. Opaque semantic fallbacks and stronger borders/text roles are required.

### Reduce Motion

Many screens already use opacity fallbacks. Remaining issues include the ungated send symbol transition, automatic Chat scrolling, long list entrance staggering, and conflating Reduce Motion with disabling all haptics. Tactile confirmation should not generally disappear solely because motion is reduced.

### Hit targets and focus

Most icon controls meet 44 points. Several text-only actions in voice capture and compact Restore actions in Backup require explicit minimum targets. Keyboard-heavy screens need scroll/focus behavior that keeps primary actions reachable.

## Brand and asset audit

- The existing `MnemoLogoMark` is a custom SwiftUI path, not an SF Symbol.
- The mark communicates a thread/continuity idea and works in monochrome.
- Its many curves can read as an arbitrary squiggle at toolbar size and should be optically simplified before any production replacement.
- `AppIcon.appiconset/Contents.json` declares universal default, dark, and tinted slots with no filenames.
- `AccentColor.colorset/Contents.json` contains no actual color value.
- `MnemoApp/Design/MnemoLogoMark.svg` is a concept/source asset, not a production app icon.

No production icon should be claimed or replaced until small-size, dark, monochrome/tinted, safe-margin, and similarity review is complete.

## Redesign priorities

1. Add an accessibility-aware semantic surface/material policy to MnemoUI.
2. Consolidate capture navigation without introducing a custom tab bar.
3. Make onboarding and all capture states scroll/focus adaptive.
4. Redesign Chat messages, composer, generation state, and source evidence.
5. Make every cited source reachable.
6. Simplify Browse filtering and row density.
7. Reorder Memory Detail around summary and original capture.
8. Return Settings to restrained native list hierarchy.
9. Add reusable previews/DEBUG fixtures and accessibility announcements.
10. Review the refined mark separately; do not replace production icon assets without approval.

## Gate A conclusion

The redesign is justified, but a custom navigation system, wholesale identity replacement, pervasive glass, or blocking launch animation is not. The next implementation phase should begin only after approval of the navigation, palette, and mark recommendations in `MnemoDesignDirection-2026-07-10.md`.
